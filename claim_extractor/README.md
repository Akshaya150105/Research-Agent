## Part 4: LLM Claim Extractor

The `claim_extractor` module is **Phase 2** of the pipeline. It takes the outputs of Parts 2 and 3 — `sections.json` from GROBID and `enriched_entities.json` from SciBERT NER — and uses a locally-served **Ollama** model (`qwen2.5`) to extract four types of structured knowledge from every section of the paper (including detached `tables.json` and `figures.json` via pseudo-section ingestion):

* **Cleaned entities** — LLM-verified, normalized entities (fixes NER tokenization noise, merges fragments, removes false positives)
* **Claims** — Structured performance, comparative, and methodological claims grounded to specific entities
* **Limitations** — Explicit weakness or constraint statements the paper admits
* **Future work** — Statements proposing directions for future research

Results are merged back into `enriched_entities.json` **and** written as a standalone `claims_output.json`.

---

### Model & Inference Backend

This module uses **Ollama** to serve the `qwen2.5` model locally over a REST API (`/api/generate`). Ollama's native `format: "json"` mode is used to constrain model output to valid JSON, and a `RateLimiter` throttles requests to avoid overloading the server.

#### `ollama_model.ipynb` — Kaggle GPU Setup

> **This notebook must be run on [Kaggle](https://www.kaggle.com/) to access a free GPU.**

`ollama_model.ipynb` automates the full GPU-backed Ollama setup on Kaggle:

1. Installs Ollama and pulls the `qwen2.5` model
2. Starts the Ollama server in the background
3. Exposes it via an **ngrok tunnel** so the CLI running on your local machine can reach it using `--ollama-host <ngrok-url>`

This is the recommended workflow when running the extractor on many papers or large documents, since `qwen2.5` inference is significantly faster on GPU.

### Setup

```bash
pip install -r claim_extractor/requirements_claim.txt
```

Requirements: `requests` (standard Ollama HTTP calls — no additional SDK needed)

> **Note:** The `requirements_claim.txt` currently lists `google-genai>=0.8.0` as a legacy entry from a previous Gemini-based version. It is not used by the current Ollama backend and can be ignored or removed.

### Usage

#### Single paper (local Ollama)

```bash
# Minimum required: grobid_parser output folder and ner_pipeline results folder
python -m claim_extractor.cli --grobid-dir path/to/grobid_output/ --ner-dir path/to/ner_results/
```

#### Single paper (remote Ollama via Kaggle ngrok tunnel)

```bash
python -m claim_extractor.cli \
  --grobid-dir path/to/grobid_output/ \
  --ner-dir path/to/ner_results/ \
  --ollama-host https://<your-ngrok-url>.ngrok-free.app
```

#### Batch mode (many papers at once)

```bash
# Recursively walks --grobid-dir for folders matching grobid_output/sections.json
# and matches them to sibling ner_results/ folders
python -m claim_extractor.cli --grobid-dir path/to/all_papers/ --batch
```

---

### CLI Options

| Flag | Short | Default | Description |
|---|---|---|---|
| `--grobid-dir` | `-g` | *(required)* | Folder containing `sections.json` from `grobid_parser` |
| `--ner-dir` | `-n` | same as `--grobid-dir` | Folder containing `enriched_entities.json` from `ner_pipeline` |
| `--ollama-host` | | `OLLAMA_HOST` env var or `http://localhost:11434` | Ollama base URL (local or remote ngrok tunnel) |
| `--rpm` | | `12` | Requests per minute throttle |
| `--batch` | | `False` | Batch mode: process all papers under `--grobid-dir` |

### Input

The pipeline reads multiple files generated by preceding extraction layers and weaves them together prior to LLM injection:

| File | Source | Description |
|---|---|---|
| `sections.json` | `grobid_parser` | Full text of each paragraph in the paper |
| `tables.json` / `figures.json` | `grobid_parser` | Decoupled media data synthesized as pseudo-sections to bypass prompt limits |
| `enriched_entities.json` | `ner_pipeline` | Model entities grouped by section, used as NER hint context for LLM verification |

### Output

#### `claims_output.json` (standalone)

A clean, self-contained output file written next to `enriched_entities.json`. The existing NER output is **extended** with new LLM-extracted fields:

```
llm_entities           — deduplicated, normalized entity list
llm_entity_index       — entities grouped by type and lowercased name
claims                 — all extracted claims
limitations            — all limitation statements
future_work            — all future work statements
llm_results_by_section — per-section raw LLM output
llm_extraction_summary — run statistics (model used, elapsed time, counts)
```
